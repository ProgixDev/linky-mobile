# Runbook — Release

Trunk is always releasable; releasing = choosing a commit and promoting it.

## OTA (JS-only changes) — continuous

Merges to `main` auto-publish an EAS Update to the **preview** channel via
the deploy-preview workflow (fingerprint-aware: if native changed, it builds
instead). Promote to production OTA:

```bash
npx eas update --channel production --message "<changelog line>"
```

Only JS/asset changes ship OTA. The fingerprint runtime policy guarantees an
incompatible update can't reach a wrong binary.

## Store release

1. Confirm `main` is green (verify + e2e + agentic QA report reviewed).
2. Bump `version` in `app.config.ts` (semver, user-facing) — PR titled
   `chore(release): v0.2.0`.
3. Tag: `git tag v0.2.0 && git push --tags` → `release.yml` runs
   `eas build --platform all --profile production --auto-submit`
   (EAS auto-increments build numbers; submission uses stored credentials).
4. Stores: complete review notes in App Store Connect / Play Console;
   changelog comes from the generated release notes (Conventional Commits).
5. Post-release: watch crashes/ANRs for 24h; any hotfix that is JS-only goes
   OTA to `production`, native hotfixes repeat from step 2 with a patch bump.

## Rollback

- OTA: `npx eas update:republish --channel production --group <previous-group-id>`.
- Store: raise the previous build in the store console (phased release pause),
  then fix forward.

Record every release in the GitHub Release auto-generated by `release.yml`.
